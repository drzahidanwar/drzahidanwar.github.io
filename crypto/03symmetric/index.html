Symmetric Cryptography

Working of Symmetric Cryptography and properties of interest.

Symmetric Cryptography

Symmetric cryptography is also known as secret key or single key cryptography. In symmetric cryptography, both parties involved in the communication share the same key. In the example (below) Alice is taking her plain-text $M$, encrypting it with a secret key $k$ and transmitting the resulting ciphertext $C$. Bob is receiving that same ciphertext $C$, decrypting it with the same secret key $k$ and retrieving the same plain-text $M$ that Alice encrypted. The overall operation can be mathematically be represented as follows:

$$E_k(M) = C$$ $$D_K(C) = M$$

This is the reason this operation is termed symmetric because the two keys are exactly the same. This is in stark contrast to asymmetric encryption where the encryption and decryption keys are different.

Symmetric Key Cryptography Scenario

In some types of symmetric encryption, the encryption key and decryption key may be different. However, it is possible to easily derive one key from another. In the example above if Alice gave the encryption key to Bob, he can easily derive the decryption key or vice versa. Although the two keys are not the same one can easily derive one from the other. This still counts as a symmetric encryption algorithm.

Another way to think about symmetric encryption is that the encryption and decryption functions are inverse operations because the same key is being used on both sides.

$$D_k (E_k(M)) = M$$

Important Properties

There are several properties of symmetric encryption that we will touch upon.

Speed

Theoretically, it is widely accepted that symmetric encryption is fast. Roughly symmetric encryption is a thousand times faster than asymmetric encryption with the same level of security.

Scalability

The major problem with symmetric encryption revolves around key distribution issues. Consider that we have two parties A and B that want to communicate securely. When using a symmetric key scheme they need to have a secret called $K_{AB}$ which is a secret key shared between A and B. The problem stems from the fact that both sides need to have a copy of this key. $K_{AB}$ needs to be communicated using an external channel which is secure to avoid eavesdropping attacks. This is achieved using key distribution algorithms to be introduced later in the course. Consider now that we have a third party C (depicted in the left side of the figure below). C needs to communicate with both A and B. For this purpose she will need two keys: $K_{AC}$ and $K_{BC}$. Now A needs a new key which is $K_{AC}$ and B needs a new key which is $K_{BC}$. So far so good. Everyone needs only two keys. What about now we have D (depicted on the right side in the figure below). If D needs to talk to A, B and C then everybody needs 3 keys.

Exponential increase in the number of keys based on number of communicating entities (figure left 2 keys per person and figure right 3 keys per person).

Let us generalize this problem to see how many total are needed for $N$ people. Everyone has $N-1$ keys and every key is shared by two persons only which gives us the total keys required as $\frac{N(N-1)}{2}$. If $N$ is really large it will require $\approx \frac{N^2}{2}$.

Group Communication

There exist many scenarios where a user needs to communicate securely to not just one user as in the case discussed above but to a group of users. Common examples include podcasts, video conferencing, remote corporate meetings and social media groups. For example, consider the figure below where A wants to send the same message to both B and C. A needs to encrypt the msg with $K_{AB}$ and then encrypt the same message with $K_{AC}$. So now what we are facing is that if one wants to send a message to $X$ people we need to encrypt $X$ copies of the same message with different keys which is obviously not a very scalable approach.

Sending a message to X parties using symmetric cryptography requires encrypting X copies of the message each with a different key.

Size Relation between plaintext and ciphertext

There exists a one to one mapping in symmetric cryptography between the plaintext and ciphertext. The ciphertext usually has the same or longer length as the plaintext. It is very hard for the ciphertext to be shorter than the plaintext unless one is conducting some sort of compression on the information.

Input Type

We have established that a block of plaintext which is meaningful for example an English sentence such as "Attack at dawn" can be encrypted using symmetric cryptography. However what if a user generates a random sequence of characters without any meaning and provides that as input to the encryption algorithm? Will that random sequence be encrypted? The answer is yes. The encryption is applied as a result of an algorithm or a procedure. As long as the inputs meet the requirements of the algorithm it should work.

For example, the DES symmetric cipher requires the input to be in chunks of 8-byte blocks. DES does not really care if it is a meaningful sentence like “I love you” which is 8 characters if you don’t count the space or it is something incorrect like “Chewbacca is mooing”. DES will simply encrypt the message. Moreover, if the correct key is applied during the decryption the original content should be properly recovered. We emphasize this point here because later when we are dealing with key distribution procedures we will see that this property can lead to malicious attacks.
